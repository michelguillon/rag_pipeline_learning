## What we're building

A document Q&A system that ingests a CV, stores it in ChromaDB, retrieves
relevant chunks on query, and passes them to Mistral for answer generation.

The pipeline runs as separate scripts with clean separation of concerns:

```
analyse.py        → inspect document structure, call Mistral for chunking recommendation
review_chunks.py  → preview proposed chunks, human approves before any embedding
ingest.py         → chunk + embed + store in ChromaDB (run once per document)
query.py          → embed query + retrieve + generate answer (run any time)
```

The full data flow:

```
Document
   └─► analyse.py ──► Mistral API ──► Recommended config (printed)
                                              │
                                       Human approves
                                              │
                                       config.json
                                              │
   └─► review_chunks.py ──► Preview chunks (printed, no embedding yet)
                                              │
                                       Human confirms
                                              │
   └─► ingest.py ──► Chunker ──► mistral-embed ──► ChromaDB (persistent)
                                              │
   └─► query.py ──► embed query ──► ChromaDB ──► Mistral ──► Answer
```

## Environment

**Development:** Windows, Docker Desktop, PowerShell
**Target deployment:** Home server (arriving Wednesday) — same Docker image, zero changes needed

### Why Docker (not venv)

venv and Docker solve the same problem — isolated, reproducible dependencies — at different levels. venv is Python-level isolation on your local machine. Docker is system-level isolation: a complete Linux environment with its own Python, packages, and filesystem. Nothing touches Windows except the files explicitly mounted as volumes.

Choosing Docker from the start means:
- No retrofitting later when moving to the home server
- ChromaDB data persists in a named volume across container restarts
- Anyone cloning the repo gets a working environment with one command
- Closer to how this would run in a real enterprise deployment

**Architecture: one container, not two.** ChromaDB runs as an embedded Python library (not a server), so there's no need for a separate ChromaDB service in Docker Compose. One `pipeline` service, one volume mount for the data directory.

```yaml
# docker-compose.yml
services:

    environment:
      - MISTRAL_API_KEY=${MISTRAL_API_KEY}

```

## Architecture decisions

### Decision 1 — Document parsing

**First read (superseded — kept for the reasoning trail):**
> The CV is a single 5-column table (26 rows). Read column 0 only — the other
> 4 columns are layout duplicates. Detect structure by paragraph style:
> `Heading 3` → company or job title; `List Paragraph` → bullet.

**What the actual document showed (Phase 2 probe, verified against raw XML):**

The first read assumed more style discipline than the document has — a common
real-world trap. The probe found:

- **It is a one-column table, not five.** Every row is a single cell with
  `gridSpan=5`; `python-docx` returns that one cell five times. Reading
  `cells[0]` works — because there is only one cell per row, not because we
  discard four duplicates.
- **Zero document-level paragraphs** with content — confirmed; the CV is
  entirely table-based.
- **Hierarchy is encoded inconsistently.** The author mixed Word heading
  *styles* with direct font-size formatting. Paragraph style alone is not a
  reliable signal — the decode needs three signals together:

| Role | Reliable signal | Renders at |
|------|-----------------|-----------|
| Section header | `Heading 1` style | 16 pt |
| Company / sub-section header | direct **14 pt** size override (style varies: Normal / Heading 3 / Heading 4) | 14 pt |
| Job title | `Heading 3` style, no size override | 11 pt |
| Bullet (achievement) | has a `numPr` element | 11 pt |
| Body text | `Normal`, no `numPr`, not 14 pt | 11–12 pt |

- **The trap:** "Microsoft" uses `Heading 3` — the same style as job titles.
  Style alone cannot separate them; the 14 pt override does.
- **Bullets:** detect by `numPr`, NOT by the `List Paragraph` style — bullets
  appear under both `List Paragraph` and `Normal` (numId 6/7/9).

**Decode rule (ordered):** `numPr` → bullet; rendered size 14 pt → company /
sub-section; `Heading 3` → job title; `Heading 1` → section; else → body.

**Parsing approach:**
- `python-docx`, walk `doc.tables[0]`, read `cells[0]` of each row.
- Do NOT hardcode the size/style thresholds. `analyse.py` *profiles* the
  document (see Decision 3) and the decode mapping is confirmed per-document —
  so the pipeline generalises to other document types, not just this CV.

### Decision 2 — Chunking strategy

We build **two collections** to run as a controlled experiment.

#### Option A — One chunk per role

Each chunk = one complete job entry (all bullets for that role combined).

```
Chunk example:
  company:  Utiq
  title:    Head of Solution Consulting
  dates:    Mar 2024 – present
  text:     "Joined at a pre-commercial stage to define and establish
             the UK solutions consulting function...
             Worked directly with C-level clients...
             Activated initial pilot campaigns..."
```

**Why this works:** Role-level chunks preserve full context for each
position. A query like "what did he do at Utiq?" retrieves the entire
entry rather than a fragment.

**Risk:** A query about a specific achievement competes with everything
else in that role's chunk. The embedding has to average across all bullets,
which dilutes the signal for precise queries.

#### Option A.2 — One chunk per bullet, with context prefix

Each bullet becomes its own chunk, but carries a mandatory prefix so it
remains self-contained.

```
Chunk example:
  prefix:  [Utiq | Head of Solution Consulting | Mar 2024–present]
  text:    "Activated initial pilot campaigns across multiple publisher
            partners, achieving 40% incremental audience reach and
            informing the broader commercialisation roadmap."
```

**Why the prefix is mandatory:** Bullets in isolation have no anchor to
the company or role. Without the prefix, a query like "what did he achieve
at Utiq?" would not retrieve a bullet that never mentions "Utiq" by name.
The prefix is not decoration — it is load-bearing for retrieval.

**Risk:** Bullets are semantically narrow. A question requiring synthesis
across a role (e.g. "summarise his responsibilities at Microsoft") will
retrieve fragments, not the full picture.

**Why we test both:** The experiment reveals whether retrieval precision
(A.2) or retrieval context (A) produces better answers for this type of
document. Neither is obviously correct — the right answer depends on the
query distribution, which we'll discover empirically.

**Expected chunk counts (approximate):**
- Option A: ~7 chunks (one per role + education + other sections)
- Option A.2: ~20–25 chunks (one per bullet across all roles)

### Decision 3 — Document analyser with Mistral recommendation

Before chunking, `analyse.py` runs a structural analysis of the document
and calls Mistral to recommend a chunking strategy.

**Why this exists:**
- Forces explicit reasoning about document structure before coding it
- Makes the pipeline reusable for other document types
- Demonstrates a meta-pattern in interviews: AI reasoning about AI pipeline
  configuration

**Human-in-the-loop by design:**
The analyser prints a recommended config but does not apply it
automatically. The human reviews and confirms before ingestion begins.
This is a legitimate production pattern for enterprise RAG where document
types vary and automatic misconfiguration is costly.

**What the analyser detects:**
- Document structure type (paragraph-based, table-based, mixed)
- A **formatting-fingerprint profile**: every distinct combination of
  (paragraph style, rendered font size, bold, list-or-not) with occurrence
  counts and a sample line — the document-agnostic core of the analyser
- A **consistency report**: objective flags where visual hierarchy is NOT
  driven by styles (bullets under multiple styles; prominent text carrying no
  heading style; heading styles overridden by direct sizing)
- Detected sections (via heading styles) with text and bullet counts
- Estimated chunk counts for Option A and Option A.2
- Average content length per section

Why fingerprints, not hardcoded rules: profiling rather than assuming makes the
analyser reusable across document types (CV, RFI, RFP), not just this CV.

**What it asks Mistral:**
Given the structural analysis above, Mistral recommends:
- Which chunking strategy fits this document type
- Where the natural chunk boundaries are
- What metadata to attach to each chunk
- Any risks or edge cases to watch for

**Output:** A `config.json` file that `review_chunks.py` and `ingest.py` read.

### Decision 4 — Vector store

**Choice: ChromaDB in persistent mode**

```python
import chromadb
client = chromadb.PersistentClient(path="./chroma_db")
```

**Why persistent (not ephemeral):**
We have multiple collections that must coexist. Ephemeral mode loses all
collections on every restart — unacceptable when running experiments across
Option A, Option A.2, and multiple distance metrics.

**Why ChromaDB (not Pinecone, Weaviate, pgvector):**
- Zero setup, runs locally
- Python-native, no external service to manage
- Persistent client writes to `./chroma_db` directory — portable via Docker
- Sufficient for this corpus size (~25 chunks max)

**Production upgrade path:**
ChromaDB → Pinecone when: corpus exceeds ~1M vectors, multi-tenant access
required, or managed infrastructure is preferred. The abstraction layer
in `ingest.py` means this swap is a one-file change.

**Docker:** ChromaDB data directory mounted as a Docker volume so it
survives container restarts and can be backed up.

**Collection naming convention (baked in from creation — cannot change metric after):**
```
cv_role_cosine     ← Option A, cosine distance
cv_role_l2         ← Option A, L2 distance
cv_bullet_cosine   ← Option A.2, cosine distance
cv_bullet_l2       ← Option A.2, L2 distance
```

### Decision 5 — Distance metric

We test two metrics. All three options are documented here so the
reasoning for exclusions is explicit.

#### Cosine similarity ✅ — tested
Measures the **angle** between two vectors regardless of their magnitude.
Two documents expressing the same idea with different amounts of text
will still land close in vector space because cosine is magnitude-independent.
Standard choice for text similarity.

Formula: `cos(θ) = (A · B) / (|A| × |B|)`
Range: -1 (opposite) to 1 (identical). ChromaDB reports as distance (1 - similarity).

#### L2 (Euclidean) distance ✅ — tested
Measures the **straight-line distance** between two points in vector space.
Sensitive to magnitude. On normalised vectors (where all magnitudes = 1)
the divergence from cosine is smaller but not zero — closely ranked chunks
can flip order, which is what makes this worth testing.

Formula: `L2 = √(Σ(Aᵢ - Bᵢ)²)`

**Why L2 can produce different rankings than cosine on normalised vectors:**
Even with magnitude = 1, the geometric relationship between distance and
angle is non-linear. Pairs of vectors that cosine ranks similarly may have
a different L2 ordering, particularly when similarity scores are close.
The experiment is designed to surface whether these ranking differences
affect answer quality in practice.

#### Inner product (dot product) ❌ — excluded
When vectors are L2-normalised (magnitude = 1), inner product equals
cosine similarity exactly: `A · B = |A||B|cos(θ) = cos(θ)` when |A|=|B|=1.

`mistral-embed` produces L2-normalised vectors (confirmed in learning
notes from Experiment 6). Testing inner product would confirm the maths
but produce identical retrieval rankings to cosine — no new information.
Excluded on those grounds, not because it's inferior.

**ChromaDB metric setting (set at collection creation, immutable):**
```python
collection = client.create_collection(
    name="cv_role_cosine",
    metadata={"hnsw:space": "cosine"}  # or "l2"
)
```

### Decision 6 — top_k

top_k is calibrated to collection size, not set as a global constant.
The guiding heuristic: top_k should not exceed ~40% of collection size,
beyond which retrieval collapses into near-full-collection return.

**The three-way tradeoff top_k sits inside:**
```
Retrieval coverage  ←──── top_k ────►  Token cost + Noise
```
More chunks = better coverage of multi-part answers, higher prompt token
cost, higher risk of irrelevant content confusing the model. In production,
token budget is usually the binding constraint, not retrieval quality alone.

#### Option A (~7 chunks):
- **Min: top_k = 1** — forces the system to commit to a single best match.
  Reveals whether the top result alone is sufficient and exposes cases where
  the right answer is in the second or third chunk.
- **Max: top_k = 3** — 3/7 = 43%, approximately the 40% ceiling.
  Sufficient to cover multi-role queries without returning the whole collection.

#### Option A.2 (~22 chunks):
- **Min: top_k = 1** — same reasoning as above.
- **Max: top_k = 5** — 5/22 = 23%, more conservative than the 40% ceiling.
  Bullet chunks are semantically narrow, so more chunks are needed to
  reconstruct a complete answer — but token budget caps the ceiling.
  Note: this is where token budget overrides the pure retrieval heuristic.

**Why we skip the middle value:**
The min/max comparison reveals the failure modes (too little context vs.
too much noise). The middle value is where you'd land in production — but
for learning, the boundaries teach more than the compromise.

### Decision 7 — Embedding model

**Choice: `mistral-embed` (fixed)**

Key properties (from Experiment 6 in learning notes):
- Output: 1024-dimensional vectors
- Vectors are **L2-normalised** (magnitude ≈ 1.0)
- Consequence: cosine similarity = dot product for these vectors
- Batch input: list of strings, output: one vector per string in order

**The one sticky choice in the pipeline:**
The embedding model is the only component that cannot be swapped freely.
Changing it means re-embedding (re-indexing) the entire corpus because
vectors from different models live in incompatible vector spaces.
The generation model, prompt, and top_k can all change without re-indexing.

**Same model for documents and queries — always:**
Retrieval works by comparing query vector to document vectors. If they
were embedded by different models, the comparison is meaningless — the
vectors exist in different geometric spaces.

---

### Decision 8 — Generation model

We test two models against the same retrieved chunks.

**Default: `mistral-small-latest`**
Fast, cheap, good for extraction and summarisation. CV Q&A is not a hard
reasoning task — the model needs to read context and report, not reason
across multiple inference steps.

**Comparison: `mistral-large-latest`**
Best reasoning quality, ~10x the cost of small. The experiment question:
does the quality improvement justify the cost for simple document Q&A?
Expected finding: marginal difference for straightforward CV queries,
more meaningful difference for synthesis or multi-hop questions.

**Temperature: 0.1 (fixed)**
Near-deterministic. Factual Q&A requires consistency, not creativity.
High temperature increases hallucination risk when the model should be
anchored to retrieved context. (Established in Experiment 3.)

### Decision 9 — Prompt design

#### Instruction (fixed — not an experiment variable)

```
You are a precise assistant answering questions about a CV.
Answer using ONLY the information in the context provided below.
If the answer is not present in the context, respond with exactly:
"I cannot find this in the provided document."
Do not use your general knowledge. Do not infer or extrapolate.
```

**Why the exact fallback phrase matters:**
It gives us a testable hallucination signal. Any response that isn't
that exact phrase for an out-of-scope question means the model drew on
training data. This makes the stress test (Hours 8–10) measurable.

#### Context format (experiment variable)

Two formats tested with the same retrieved chunks:

**Flat:**
```
Context:
[chunk text]
[chunk text]
[chunk text]

Question: ...
```

**Labelled:**
```
Context:
[Source: Utiq | Head of Solution Consulting | Mar 2024–present]
[chunk text]

[Source: Microsoft | Senior Product Manager | Apr 2022–Mar 2024]
[chunk text]

Question: ...
```

**Why labelled may outperform flat:**
The model can attribute answers to specific roles, reducing confusion when
similar content appears across multiple positions. Also makes hallucination
easier to detect — a wrong source citation is a visible error.

### Step 1 — Document analyser (`analyse.py`)

### Step 2 — Chunk reviewer (`review_chunks.py`)

> **Naming note:** this step was originally specced as `inspect.py`. That name
> is unusable — `inspect` is a Python stdlib module, and a file by that name in
> the project root shadows it for *every* script (chromadb imports stdlib
> `inspect` on its first import line). Renamed to `review_chunks.py` in Phase 1.

**What it does:**
1. Reads `config.json`
2. Parses the document using the configured strategy
3. Prints every proposed chunk to terminal — no embedding, no storage
4. For each chunk shows: index, section type, metadata, full text, word count
5. Prints summary: total chunks, avg word count, min/max word count

**Why this step exists:**
Embedding costs tokens and writes to ChromaDB. Inspecting first means
you catch bad chunk boundaries before spending API budget on them.
In production this is the equivalent of a data validation step before
a batch job.

```
solutions consulting function, including operating model...
[full text]
─────────────────────────────────────────

Proceed to ingestion? (y/n):
```

### Step 3 — Ingestion (`ingest.py`)

**What it does:**
1. Reads `config.json`
2. Chunks the document according to configured strategy
3. Creates 4 ChromaDB collections (or skips if already exist):
   - `cv_role_cosine`, `cv_role_l2`
   - `cv_bullet_cosine`, `cv_bullet_l2`
4. Embeds all chunks via `mistral-embed` (batched, with retry)
5. Stores chunks + embeddings + metadata in each collection
6. Prints ingestion summary per collection

**ChromaDB client:**
```python
import chromadb

```

**Collection creation:**

```python

    "company": "Utiq",

```

### Step 4 — Query pipeline (`query.py`)

**What it does:**
1. Accepts a question and experiment parameters via CLI
2. Embeds the question using `mistral-embed`
3. Queries the specified collection with the specified top_k
4. Assembles the prompt (flat or labelled context format)
5. Calls Mistral for generation
6. Prints: retrieved chunks with similarity scores, full prompt, answer,
   token usage
7. Optionally saves result to `outputs/` as JSON

**CLI interface:**
```powershell
python query.py "What did he do at Microsoft?" `
  --collection cv_role_cosine `
  --top-k 3 `
```
